# 강남구 토폴로지 분석 — 문제 진단 · 시각화 방법론 · 대책

> 분석일 2026-05-22 · 대상 `gangnam_topology.json` / `gangnam_speed_data.csv`
> 산출물 폴더: `output/22_1356_GN_RLMetod/`
> `gn_fullmap.png` · `gn_zoom_infeasible.png` · `gn_pixel_compare.png` ·
> `gn_camera_pan.gif` · `gn_diagnosis.png` · `_analysis.json`

---

## 요약

강남구 실도로망(1995 노드)을 RL 최소연료 경로탐색에 쓰려고 `clean` 버전을
만들었으나, 학습·실험에서 **도달률 0%**가 지속됐다. 본 분석은 그 원인을
코드 레벨로 규명한다. 결론: `clean` 작업은 토폴로지를 **로드 가능**하게만
만들었을 뿐, **주행 가능(traversable)**하게 만들지 못했다. 핵심 원인은
**전 노드의 93%에서 좌회전이 금지**되어 있어 Dijkstra 최단경로조차 실행
불가능(infeasible)하다는 점이다.

---

## 1. clean 버전 — 생성 이유와 한계

### 1.1 왜 만들었나 · 무엇이 달라졌나

원본 `gangnam_topology.json` (1995 노드 · 2529 링크)은 `RoadNetworkEnv` 로
바로 쓸 수 없었다.

| 원본의 문제 | 내용 | clean 의 조치 |
|---|---|---|
| 댕글링 링크 | 링크 끝점 90개가 `nodes` 목록에 없음 → 로드 시 `KeyError` | 양 끝점이 모두 존재하는 링크만 유지 (2529 → **2439**) |
| 길이 단위 불명 | `len` 값이 0.01~2.39 — 실은 km 인데 미문서화, 타 토폴로지(6x6·12x12)는 m 단위 | 노드 위경도로부터 **haversine 재계산 → m 단위** 통일 |
| 속도 CSV 포맷 | GIS 원본은 `07:00`…`08:55` 컬럼 + 도로 메타 수십 개 | env 포맷 `link_id, t_0…t_23` 으로 변환 |

→ **그렇게 한 이유**: 다중 토폴로지 학습(6x6_cross·10x10·강남구 동시 학습)에서
`RoadNetworkEnv` 가 강남구를 **로드**할 수 있고, 길이 단위가 타 토폴로지와
**정합**하도록.

### 1.2 그럼에도 안 되는 이유 — 코드 레벨 진단

clean 은 **로드 가능성**만 해결했고 **주행 가능성**은 해결하지 못했다.
(`gn_diagnosis.png` · `gn_fullmap.png` 참조)

**① 좌회전 금지가 전 교차로의 93%** — 결정적 원인

- `gangnam_clean_topology.json` 의 노드 중 `left_turn_allowed=true` 는
  **138개 / 1995개 (6.9%)** 뿐. 나머지 **1857개(93.1%)는 좌회전 금지**.
- `environment.py` 의 `get_valid_actions()` 는 `left_turn_allowed=false`
  노드에서 좌회전 이동을 **행동 공간에서 제거**한다(`_node_allows_left`).
- 결과: Dijkstra 가 제약 없는 무방향 그래프에서 찾은 최단경로가
  좌회전 금지 노드를 지나가면 에이전트가 그 경로를 **따라갈 수 없다**.

**② Dijkstra 최단경로 자체가 infeasible solution**

`start=111357 → goal=647033` 의 Dijkstra 최단경로:

| 항목 | 값 |
|---|---|
| 경로 | 26 링크 · 5332 m · 27 노드 |
| 중간 노드 25개 중 실행 가능 | 20 (80%) |
| **좌회전 금지 위반** | **5 (20%)** |
| 방위 top-4 절단 위반 | 0 |

→ 최단경로의 **20%가 통행 불가**. (`gn_zoom_infeasible.png` — 5개 위반 지점
   줌인)

**③ 실제 주행 — 도달률 0%**

`ShortestDijkstra` 로 30회 주행:

| 항목 | 값 |
|---|---|
| 도달률 | **0% (0/30)** |
| 평균 스텝 | 14 / 200 (max_steps) |
| 종료 노드 | 30회 전부 `338773` 에서 정지 |

에이전트는 최단경로를 따라가다 좌회전 금지 노드에서 대체 행동
(`valid_actions[0]`)으로 이탈하고, 14스텝 만에 노드 `338773` 에 도달한다.
`338773`(차수 3·무신호·좌회전 금지)에서는 진입로(U턴)를 뺀 나머지 2개
간선이 모두 좌회전이라 **`get_valid_actions()`가 빈 리스트** → 완전 정지.
좌회전 금지가 만든 **"행동 불능 함정 노드"**의 전형. (`gn_fullmap.png` 의
주황 점선 = 이탈 경로)

**④ 막다른 노드 과다 — 차수 1 노드 397개 (20%)**

차수 분포: degree1 **397** · degree2 570 · degree3 785 · degree4 230 ·
degree5 12 · degree6 1. 도로망 추출 과정의 경계 절단·미병합 아티팩트로,
전체의 20%가 막다른 길이다. 무작위 start/goal 학습이 실패하는 또 다른 이유.

**⑤ 일방통행(one-way) 정보 손실**

원본·clean 토폴로지의 링크 필드는 `id·end1·end2·len·LANES` 뿐 —
**방향 정보가 없다.** `RoadNetworkEnv` 는 링크를 무방향으로 적재한다.
강남구 GIS 속도 CSV에는 `ONEWAY` 컬럼이 있으나 토폴로지로 옮겨지지 않았다.
→ 시뮬레이터 안에서 "통행 가능"한 경로도 현실에서는 **일방통행 역주행**일
수 있어, 해(解)가 **현실적으로 infeasible** 하다.

**연결성은 문제 아님** — start 에서 1985/1995(99.5%) 도달 가능, goal 도
같은 컴포넌트. 즉 *연결*은 되어 있으나 *주행 규칙(좌회전·방향)* 때문에
실제로는 갈 수 없는 **traversability** 문제다.

## 2. 강남구 시각화 방법론

### 2.1 기존 방법론의 한계

`simulation.py` 는 5~7 km 도로망 전체(1995 노드·2439 링크)를 1296px·**72dpi**
한 프레임에 렌더한다. 노드가 sub-pixel 크기가 되어 구조 파악이 불가능하고,
래스터 이미지라 **확대하면 픽셀 열화**로 경로를 식별할 수 없다.
(`gn_pixel_compare.png` 좌측 — 72dpi 확대 시 형체 불명)

### 2.2 제안 — 고해상도 시각화 방법론 (MVP: `util/gangnam_hires_viz.py`)

| 기법 | 내용 |
|---|---|
| **미터 투영** | 위경도를 등거리 평면 좌표(m)로 투영 → 왜곡 없는 비율 |
| **자유 확대** | `render_window(center, half_width)` 가 임의 영역을 **프레임 픽셀 전부**로 직접 렌더 → 어느 배율에서도 선명(벡터 품질). 전체를 한 번 렌더한 뒤 확대하는 게 아니라, 보고 싶은 영역만 새로 렌더 |
| **카메라 자동 이동** | 경로를 따라 카메라 중심이 이동/줌하는 GIF — 각 프레임이 좁은 영역(반경 240 m)을 풀 해상도로 |
| **고DPI 정적 산출물** | 전체도 200 dpi + 통행불가 지점 줌 인셋 |

### 2.3 MVP 산출물 — 1의 문제를 가시화

- **`gn_pixel_compare.png`** — 동일 영역을 (좌) 기존 72dpi 확대 vs
  (우) 본 방법론 직접 고해상도 렌더. 픽셀 열화 해소를 직접 비교.
- **`gn_camera_pan.gif`** — 1단계: 전체망 → 출발점 줌인(자유 확대) /
  2단계: Dijkstra 경로를 따라 카메라 자동 이동, 좌회전 금지 지점 통과 시
  "통행 불가" 경고 표시.
- **`gn_zoom_infeasible.png`** — 좌회전 금지 5개 위반 노드 줌 인셋 —
  고해상도라 교차로 형상까지 선명.
- **`gn_fullmap.png`** — 전체 도로망 + Dijkstra 경로 + 통행불가 지점 +
  막다른 노드 + 에이전트 이탈 경로.

→ 기존 방법론으로는 보이지 않던 §1 의 구조적 문제(좌회전 함정·막다른 길)가
   본 방법론에서 명확히 드러난다.

## 3. 대책

### 3.1 §1 문제 해결 방법론

| 문제 | 대책 |
|---|---|
| 좌회전 금지 93% | 핵심. `left_turn_allowed` 단일 플래그가 과도하게 false. ① 무신호 교차로는 비보호좌회전 허용(`true`)으로 보정 ② 근본책 — 실제 회전제한(turn-restriction) 데이터를 확보해 노드 단일 플래그가 아닌 **(진입링크, 진출링크)별 회전 허용**으로 인코딩 |
| 일방통행 손실 | GIS 의 `ONEWAY` 를 토폴로지 링크에 방향으로 반영하고 `RoadNetworkEnv` 를 **유방향 그래프**로 |
| 막다른 노드 397개 | 추출 아티팩트. ① 방향성 반영 후 **최대 강연결요소만 유지** ② 추출 경계 확장 ③ 근접 미병합 노드 스냅 |
| 차수>4 (13개) | 경미 — 수용하거나 `K_HOP1` 상향 또는 노드 병합 |
| 검증 | 보정 후 Dijkstra·에이전트 도달률 재측정 (현재 0% → 목표 ~100%) |

### 3.2 강남구 저연료 실험에 추가로 필요한 조치

1. **주행 가능 토폴로지 확보** — §3.1 이 선결 조건.
2. **현실적 OD·커리큘럼** — 1995 노드 무작위 start/goal 은 대부분 실패.
   실제 통행 기·종점(OD) 쌍 사용, 또는 단거리→장거리 커리큘럼.
3. **보상 shaping 의 규모 적응** — 12x12 실험에서 대형맵 희소보상 탐색난
   해소에 포텐셜 기반 shaping 이 필수였음. 강남구 규모에 맞춰 적용.
4. **회전제한 인지 플래너** — `StaticFuelDijkstra` 에 회전제한 인지 로직
   적용(12x12 에서 이미 반영). 비교 기준선의 정합성 확보.
5. **신호 회피 시나리오 성립 확인** — 신호 노드가 11.6%로 희소. 저연료
   경로탐색의 효과를 보려면 후보 경로상에 신호 우호/불리 대비가 존재해야 함.
6. **연산 예산** — 1995 노드·장거리 경로의 학습 비용 산정.

### 3.3 실세계 실효성 검증을 위한 추가 자료

| 추가 자료 | 활용 방법 |
|---|---|
| 회전제한·일방통행 데이터 (GIS/OSM) | 토폴로지를 실제 주행 규칙과 일치시켜 해의 현실 타당성 확보 |
| 실측 신호 SPAT (T-DATA) | 강남구 실제 교차로 신호 현시를 반영 (12x12 에 적용한 로직 확장) |
| 실제 OD 통행 데이터 | 현실적 통행 수요로 실험 — 무작위 경로의 비현실성 제거 |
| GPS·프로브 차량 궤적 | 속도 모델 검증 + 모델이 택한 경로 vs 실제 운전자 경로 비교 |
| 실측 연료·배출 로그 | VT-Micro 연료 추정의 실측 정합성 검증 |
| 내비게이션(TMAP·Kakao) 경로 | 저연료 경로 ↔ 최단시간 경로 차이·절감량 정량 비교로 실효성 입증 |

## 4. 결론

`clean` 버전은 강남구 토폴로지를 **로드 가능**하게 만든 정당한 1차 정제였으나,
**좌회전 금지 93% · 일방통행 정보 손실 · 막다른 노드 20%** 라는 구조적 결함이
남아 Dijkstra 최단경로의 20%가 실행 불가, 에이전트 도달률 0% 에 머문다.
이는 *연결성*이 아닌 *주행 규칙* 결함이며, 회전제한·방향 데이터를 갖춘
토폴로지 재정제가 강남구 RL 실험의 선결 조건이다. 본 분석을 위해 개발한
고해상도 시각화 방법론(`gangnam_hires_viz.py`)은 이 구조적 문제를 누구나
확인할 수 있게 가시화하며, 향후 강남구 디버깅·발표의 표준 도구로 쓸 수 있다.
